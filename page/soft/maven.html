概述

Apache Maven( [`meivn] )是一个软件项目管理和依赖管理工具。基于项目对象模型（POM）的概念，Maven可以通过资源仓库管理项目的构建，报告和文件。

Maven对于项目目录结构、测试用例命名方式等内容都有既定的规则，只要遵循了这些成熟的规则，用户在项目间切换的时候就免去了额外的学习成本，可以说是约定优于配置（Convention Over Configuration）。

作为Apache组织提供的一个颇为成功的开源项目，Maven主要服务于基于Java平台的项目构建，依赖管理和项目信息管理。无论是小型的开源类库项目，还是大型的企业级应用；无论是传统的，还是流行的，Maven都能大显身手。

目录&文件 说明
src/main/java 项目JAVA源码文件存储目录
src/main/resources 项目配置文件存储目录
src/test/java 项目JAVA测试源码文件存储目录
src/test/resources 项目测试配置文件存储目录
JRE System Library 项目的JRE类库，开发工具默认提供，用于源码编译
target 项目构建（编译，打包，发布站点）后，会将结果放置在此目录中
pom.xml 项目中maven依赖核心配置文件，类似于web系统中的web.xml,非常重要

在src/main/java目录中添加com.maven.atguigu.Hello类

在src/test/java目录中添加com.maven.atguigu.test.HelloTest类

类创建好后会发现，程序无法编译通过，因为在测试类HelloTest中使用了JUnit测试库中的类org.junit.Test，但是在当前项目的编译环境中并不存在，所以编译无法通过，发生了错误

解决方法非常简单，只需要在项目中引入测试类库即可，有很多种方法可以实现。

我们这里采用Maven方式实现：在项目的pom.xml文件中增加配置

你会惊奇的发现，程序编译通过了，且项目的编译库增加了一个编译库Maven Dependencies

为什么我们没有引入测试类库，程序也能编译通过？

究其原因，是Maven工具帮我们将测试类库增加到当前项目中，就是刚刚新增加的那个编译库Maven Dependencies

在JAVA程序中，A类用到了B类，我们一般称之为A依赖于B，如果B类中用到了C类，那么我们称之为A间接地依赖于C。

Maven是如何将测试类库引入到项目中的呢，这里就需要说明之前向pom.xml文件中增加的内容了

在配置文件中<dependencies>表示依赖关系，其中<dependency>表示具体的某一个依赖的类库，如：junit

当增加<dependency>后，就表示当前项目依赖于指定的类库(jar)，那么Maven就会在自己的仓库中按照指定的规则查找这个类库(jar)。

maven根据<groupId>，<artifactId>，<version>三个标签在仓库中进行定位，查找依赖的类库


如果你的maven仓库的位置为: "d:/maven/repository/", 根据规则，maven会从"d:/mavem/repository" + <groupId> + <artifactId> + <version>路径中查找类库。


如果要查找junit类库，那么maven会从 ：“d:/maven/repository/junit/junit/4.12/” 路径中查找指定的类库（jar包）

Maven不仅仅将依赖的类库引入到项目中，连间接依赖的类库也会引入到当前项目中，这样就省掉了我们自己查找的麻烦。

为什么使用Junit执行测试的时候没有出现错误，而使用Maven做测试时，会发生错误？并且使用Maven测试Hello项目又没有问题呢？

使用Maven执行测试时，项目所依赖的类库必须从仓库中查找，即使在同一个工作空间中也无法直接关联。由于仓库中并不存在Hello的类库（jar包），所以发生错误。

创建第三个Maven项目

在项目视图中创建第三个Maven项目SayHelloWorld。并增加对项目HelloWorld的依赖

在src/main/java目录中添加com.maven.atguigu.SayHelloWorld类

在src/test/java目录中添加com.maven.atguigu.test.SayHelloWorldTest类

项目SayHelloWorld依赖于项目HelloWorld,而项目HelloWorld中依赖于junit，由于maven支持间接依赖，所以项目应该会自动依赖于junit，不会出现错误。但是实际上你会发现，情况不是这样的。

在HelloWorld项目Junit依赖关系中的 <scope>test</scope> 表示依赖类库的使用范围

在依赖声明中，scope标签指定的就是当前项目在什么情况下对目标项目有依赖。可选值包括：compile、test、provided、runtime等。

  1. compile：默认值。编译依赖范围。对于编译，测试，运行三种classpath都有效。
  2. test：测试依赖范围。只对于测试classpath有效。
  3. provided：已提供依赖范围。对于编译，测试的classpath都有效，但对于运行无效。因为由容器已经提供，例如servlet-api。
  4. runtime:运行范围，只在运行时有效

如果项目SayHelloWorld所依赖的项目HelloWorld, Hello中都依赖于Junit，但是依赖的版本不一样怎么办？

依次修改项目HelloWorld, Hello中的Junit版本，对比观察效果

当项目存在相同的依赖关系时，需要遵循路径最近者优先，路径相同，最先声明者优先的原则。

创建Maven Web项目

Maven项目默认创建的项目为java项目，可以被打包为jar文件，安装到仓库中，让其他项目使用，也可以创建为web项目，进行web服务

Spring Tool Suite 通过 Maven 建立web项目时，默认的web.xml文件版本为 2.3，jre版本为 1.5。

web.xml文件2.3版本，jre版本1.5相对于当前主流项目开发来讲版本有些过时了，如果需要变更web.xml和jre的版本，需要进行相关的配置

  1. 修改jre的版本为1.8，需要在pom.xml文件中增加配置

  2. 创建maven web项目时，可以默认采用servlet 2.5版本，将项目的打包方式，设置为war,那么当前的项目就变为web项目了。

    这种创建web项目的方式会导致项目缺少必要的web.xml文件，还需要做一些相关设置

    右键点击项目HelloWeb，选择菜单Properties Project Facets 先取消Dynamic Web Module复选框选中，应用(apply)后再重新选中后点击Further configuration available链接

    在弹出的窗口中输入生成web.xml文件的路径后点击应用即可

Maven的继承和模块化

实际的项目开发中，会在开发时将项目划分为多个不同的子项目，然后在运行时通过依赖关系将多个子项目合成一个项目后进行操作。

在这种情况下，如何保证每一个子项目所依赖的类库是相同的？当子项目过多时，如何能简化每一个子项目的测试，打包，安装的操作？

让每一个子项目所依赖的类库是相同的，可以通过创建父项目，让所有子项目继承父项目的依赖关系的方法解决。

创建父项目，需要将父项目的打包方式设置为pom方式，表明这个项目不用做开发，只用于管理

创建子项目，在配置中增加父项目关联即可

由于子项目继承了父项目，所以在子项目的pom.xml文件中，会增加相关配置信息，黄色的警告线表示该配置项已经从父项目中继承过来，不需要配置，删除即可

当子项目过多时，可以通过maven模块化的方式简化每一个子项目的测试，打包，安装等操作。

在父项目中，通过模块化的操作关联子项目，并在子项目中增加配置信息。然后可以通过父项目做多个命令操作，而同时子项目也会做相同的操作

Maven的生命周期

在第二个Maven项目中,我们执行了maven相关的命令：Maven test, Maven install, 你会发现在安装的同时也会进行测试，这说明不同的命令之间存在关联，这种关联体现了maven生命周期的概念。

Maven有三套相互独立的生命周期

  1. Clean Lifecycle: 进行真正的构建之前进行一些清理工作
  2. Default Lifecycle: 构建的核心部分，编译，测试，打包，安装，部署等等
  3. Site Lifecycle: 生成项目报告，站点，发布站点

它们是相互独立的，你可以仅仅调用clean来清理工作目录，仅仅调用site来生成站点。当然你也可以直接运行clean install site 运行所有这三套生命周期。

maven的每一个生命周期都包含很多的命令，而且命令之间存在先后顺序，如果执行顺序靠后的命令，maven会将靠前命令依次执行，这样就简化了操作的步骤，方便了很多。

Clean Lifecycle

Maven命令 说明
pre-clean 执行一些需要在clean之前完成的工作
clean 移除所有上一次构建生成的文件
post-clean 执行一些需要在clean之后立刻完成的工作

在一个生命周期中，运行某个阶段的时候，它之前的所有阶段都会被运行。也就是说，clean 等同于 pre-clean clean，如果我们运行post-clean，那么pre-clean，clean都会被运行。这是Maven很重要的一个规则，可以大大简化操作

Maven的仓库

Maven工具默认的仓库分为两类

Maven仓库 说明
本地仓库 每个用户只有一个本地仓库
本地仓库中的项目可以为本机上的所有Maven项目所共享
远程仓库 私服——是一种特殊的远程仓库，它是架设在局域网内的仓库：为局域网中全部的Maven项目共享

中央仓库——Maven默认的远程仓库：可以为全世界的Maven用户共享。中央仓库的地址是：http://repo1.maven.org/maven2

镜像——用来替代中央仓库，分担中央仓库的负荷，速度一般比中央仓库快。

Maven在仓库中检索项目, 首先在本地仓库中查找，如果找不到就到局域网中的私服中去找，还找不到就连接中央仓库或镜像。从中央仓库或镜像中找到后，如果局域网中架设了私服，就保存到私服上，下次同一局域网中的其他用户直接从私服上下载即可，不必连接中央仓库或镜像了。如果局域网中没有架设私服，则保存在本地仓库中。
